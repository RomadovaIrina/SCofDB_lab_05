## 1. Реализация Redis-кэша
_TODO: Опишите, что именно кэшируется и с какими ключами._

Нужно указать:
- ключ каталога;
- ключ карточки заказа;
- TTL;
- логику cache hit/miss.

реализован Redis-кэш для двух видов данных: каталога товаров и карточки заказа.

Для каталога используется ключ:

catalog:v1

Для карточки заказа используется ключ вида:

order_card:v1:{order_id}

Для кэшируемых данных задан TTL, равный 300 секундам. 

Логика  cache hit / cache miss:

при запросе сначала выполняется попытка чтения данных из Redis при use_cache=True;
если значение найдено, оно возвращается сразу, без обращения к базе данных (cache hit);
если значение отсутствует, выполняется запрос к бд, после чего результат сериализуется и сохраняется в Redis с TTL (cache miss).
при use_cache=false чтение выполняется напрямую из БД (без использования кэша).

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)
_TODO: Опишите шаги и результаты stale cache._

Минимум:
1. Прогрев кэша.
2. Изменение заказа в БД без инвалидации.
3. Повторный запрос из кэша.
4. Чем stale ответ отличается от данных в БД.

Последовательность действий была следующей:

Выполнялся запрос карточкb
`/api/cache-demo/orders/{id}/card?use_cache=true`
Это приводило к прогреву кэша: данные заказа загружались из БД и сохранялись в Redis.
Затем заказ изменялся через endpoint
`/api/cache-demo/orders/{id}/mutate-without-invalidation`
При этом значение total_amount в базе данных изменялось, но инвалидация кэша намеренно не выполнялась.
После этого карточка заказа повторно запрашивалась с параметром `use_cache=true`.
В результате клиент получал устаревшие данные из Redis, хотя в БД уже находилось новое значение.

конкретно в самом тесте было:
1. до изменения `total_amount==100.0`
2. после измнения в бд `total_amount==250.0`
3. достаем кэшированный ответ и получаем `total_amount==100.0`
4. но если без кэша `total_amount==250.0`

## 3. Починка через событийную инвалидацию
_TODO: Опишите механизм события и инвалидации._

Нужно указать:
- где генерируется событие изменения заказа;
- где обрабатывается событие;
- какие ключи инвалидируются и почему.

Изменение заказа выполнялось через:

`/api/cache-demo/orders/{id}/mutate-with-event-invalidation`

Здесь после обновления данных в таблице orders вызывалась инвалидация кэша через CacheService.

Событие изменения заказа происзодит в момент обновления заказа. После изменения записи в БД вызываются методы удаления соответствующих кэш-ключей.

Используются: `invalidate_order_card(order_id)`, `invalidate_catalog()`

Инвалидируются:

order_card:v1:{order_id} — потому что карточка конкретного заказа изменилась;
catalog:v1 — потому что изменение данных заказа потенциально может повлиять на агрегаты каталога.

После удаления кэш-ключа следующий запрос к карточке заказа снова вызывает чтение из БД, получает уже актуальные данные и заново прогревает кэш. Это устраняет проблему stale cache и восстанавливает консистентность.

## 4. Rate limiting endpoint оплаты через Redis
_TODO: Опишите реализацию лимитов._

Нужно указать:
- policy (например N запросов за M секунд);
- ключ лимита (по user_id/ip);
- поведение при превышении (`429`);
- какие заголовки возвращаются клиенту.

Ограничение: не более 5 запросов за 10 секунд для одного клиента.
Это задаётся параметрами :
`limit_per_window = 5`
`window_seconds = 10`

Ограничение применяется только к endpoint оплаты:

`/api/orders/{order_id}/pay`
`/api/payments/retry-demo`

Для идентификации клиента:

если в заголовках присутствует X-User-Id, то лимит считается по user_id;
если X-User-Id отсутствует, то в качестве идентификатора берётся IP.

При превышении личита:

если INCR в окне превышает лимит, endpoint возвращает 429 (too any requests)

В тесте проверялось:

- Создавалось 6 разных заказов.
- Для всех запросов использовался один и тот же заголовок X-User-Id, чтобы лимит считался для одного клиента.
- Первые 5 запросов на оплату проходили успешно и возвращали код 200.
- Шестой запрос получал код 429.
- Дополнительно проверялось, что сервер возвращает корректные заголовки:
- X-RateLimit-Limit = 5
- X-RateLimit-Remaining уменьшается от 4 до 0

Таким образом, rate limiting на Redis был реализован корректно: он ограничивает частоту обращений к endpoint оплаты, возвращает ожидаемый код ошибки при превышении лимита и сообщает клиенту текущие параметры ограничения через HTTP-заголовки.

## 6. Выводы

Рекомендуется осветить:

2. Почему инвалидация сложнее, чем кэширование.
3. Почему rate limiting полезен даже при наличии бизнес-валидаций.

Простое добавление кэша без инвалидации приводит к stale data, даже если данные в БД уже обновлены корректно.

Rate limiting отсекает всплеск повторов и защищает сервис на входе, до дорогих операций в БД.
Комбинация кэша + событийной инвалидации + rate limiting даёт одновременно производительность, консистентность чтения и устойчивость под нагрузкой.
В данной ЛР с помощью Redis удалось увидеть хорошее ускорение в несколько раз
Тем не менее redis кэш не всегда поможет, тк изменчивых данных требуется часто инвалидировать кэш поэтому нет смысла его использовать 
Redis подходит для ограничения запросов (rate limiting), что позволяется защищаться от DDoS-атак.
